# Phase 4 reference: worked example for a class with fields, a static method, and an instance method

Worked example tying together every rule that Phase 4 (`04-phase4-plan.md`) applies to a `Ty::Class` top: field rendering, constructor shape, static-method binding (`defineFunction`), instance-method binding (class-field `defineInstanceFunction(...).bind(this)`), JSDoc docstring rollup, FQN passed to the factory, and `_async` companion fan-out.

The example mirrors the canonical "class with statics + instance methods" shape from `00a-spec-codegen-mappings.md` lines 81-87 (`baml.media.Pdf.from_url` / `baml.io.File.close`), but uses a user-package class so it routes to a normal user-namespace leaf rather than the stdlib re-export early-return path.

## Input — `baml_src/lorem.baml`

```baml
// File: baml_src/lorem.baml — user package, namespace `lorem`
// Fully qualified BAML symbol: user.lorem.Resume

/// A resume parsed from raw applicant text.
class Resume {
  /// Candidate's full name.
  name string

  /// Email address, if extracted.
  email string?

  /// Tags / skills, in declaration order.
  tags string[]

  /// Years of professional experience.
  years_experience int

  /// Build a Resume from a LinkedIn profile URL.
  function from_linkedin_url(url: string) -> Resume {
    $rust_fn
  }

  /// Render a short text summary of the resume.
  function summary(self) -> string {
    "{self.name} ({self.years_experience} yrs)"
  }
}
```

## TIR view (what reaches codegen)

After `baml_compiler2_tir` normalizes the source, `sdkgen_nodejs::to_source_code` sees one `Symbol::Class` entry keyed by `user.lorem.Resume`:

- `name`: `Name { pkg: "user", namespace_path: ["lorem"], bare_name: "Resume" }`
- `docstring`: `Some("A resume parsed from raw applicant text.")`
- `generic_params`: `[]`
- `properties`:
  1. `{ name: "name", ty: Ty::String, docstring: Some("Candidate's full name.") }`
  2. `{ name: "email", ty: Ty::Optional(String), docstring: Some("Email address, if extracted.") }`
  3. `{ name: "tags", ty: Ty::List(String), docstring: Some("Tags / skills, in declaration order.") }`
  4. `{ name: "years_experience", ty: Ty::Int, docstring: Some("Years of professional experience.") }`
- `static_methods`:
  1. `Function { name: "from_linkedin_url", arguments: [{ name: "url", ty: Ty::String }], return_type: Ty::Class(user.lorem.Resume), docstring: Some("Build a Resume from a LinkedIn profile URL.") }`
- `instance_methods`:
  1. `Function { name: "summary", arguments: [{ name: "self", ty: Ty::Class(user.lorem.Resume) }], return_type: Ty::String, docstring: Some("Render a short text summary of the resume.") }`

The static-vs-instance split happens upstream — methods whose first argument is `self` land in `instance_methods`; the rest land in `static_methods`.

## Routing

`route(name, Symbol::Class) -> LeafPath` (Phase 2):
- `pkg == "user"` → no top-level prefix.
- `namespace_path == ["lorem"]` → one segment.

Result: `LeafPath { segments: ["lorem"] }`. Per the file-layout rule (`00a-spec-codegen-mappings.md:53-62`), each namespace is a directory with an `index.ts` + sibling `index.d.ts`, so the class renders into `baml_sdk/lorem/index.ts` (and its `index.d.ts` sibling — see "Companion `.d.ts` note" below).

### Things to notice

- **`from_linkedin_url` is NOT an instance member** — it's a `static` class property using `defineFunction`. Calls look like `Resume.from_linkedin_url("https://...")`. No receiver injection at the static-method site.
- **`summary` and `summary_async` are per-instance bound class fields.** Calls look like `(new Resume({...})).summary()`. Each is `defineInstanceFunction(...).bind(this)`, so the receiver is captured once at construction; a detached reference (`const f = resume.summary; f()`) still carries its receiver. (Cost: one bound-function allocation per instance method per instance.)
- **No proto-level type validation.** The class definition is purely TS-side. Wire-side validation happens in Phase 5's `proto.ts` decoder; Phase 4 emits the shape only.
- **No `BamlStream` / `BamlHandle` imports** in this example because none of the field types or method signatures touch them. If `Resume` had a `stream Stream<Resume$stream, Resume>` field, the leaf would also import `BamlStream`. If a field's type was `$rust_type`, the leaf would import `BamlHandle`.

### `.d.ts` rules

- **`export declare class`** instead of `export class`. The `declare` keyword tells TS "this is an ambient declaration — runtime code lives elsewhere." No `import { defineFunction }` / `defineInstanceFunction`, no `Object.assign`, no `.bind(this)` — those are runtime concerns the `.ts` file owns.
- **Drop `!:`.** Definite-assignment assertions are an implementation detail of how the `.ts` file constructs instances; the type contract is just `name: string`.
- **Constructor signature without a body.** Same shape as the `.ts` constructor parameter list.
- **Static methods as property declarations** of function type, mirroring the typed signature from the `.ts` assertion.
- **Instance methods as bound-member declarations** (`summary: () => string;`), NOT method syntax (`summary(): string;`). This matches the `.bind(this)`-produced shape and `00a-example-ts-codegen-type-shapes.md:438-441`. The synthetic `self` never appears in the surface type.
- **No private members.** `.d.ts` is the public type surface.
- **JSDoc preserved.** All class- and member-level JSDoc carries through verbatim — that's the whole point of having the `.d.ts` (tooling can hover-show the docs from either file).

## Companion `.d.ts` rationale

Both files are emitted: the `.ts` carries the runtime + types; the `.d.ts` carries the slim public type surface for consumers and tooling. See `04-phase4-plan.md` §"Output file shape per leaf" and §".d.ts content rules" for the full rule list.

In short: the `.d.ts` strips every runtime detail — `defineFunction(...)` / `defineInstanceFunction(...)` calls, `Object.assign`, `.bind(this)`, the `as (...) => ...` type assertions — and leaves only the typed shape consumers care about. Hover-docs and "go to type definition" read from the `.d.ts`; the `.ts` is what actually executes.
